### 评测流程

```
locomo10.json
     │
     ▼
┌──────────────────────────────────┐
│  run_locomo_official.py          │
│                                  │
│  1. 清空记忆文件                  │
│  2. 写入对话为记忆文件（直接写盘） │
│  3. 逐题清空会话 + 发送问题       │──── WebSocket → iceCoder Harness
│  4. iceCoder 召回记忆并回答       │     （记忆召回 → LLM 生成）
│  5. Judge 评分（并行）            │──── DeepSeek API
│  6. 汇总统计                     │
└──────────────┬───────────────────┘
               │
               ▼
      result_official.json
               │
               ▼
      report_official.md
```

### 关键设计决策

| 决策 | 原因 |
|------|------|
| 对话直接写入记忆文件 | 绕过 WebSocket → LLM Harness 循环，注入速度从数小时降至秒级 |
| 每个 QA 前清空会话 | 确保模型依赖记忆召回而非对话历史，测试更纯粹 |
| Judge 并行评分 | 4 线程池后台打分，与下一个 QA 的 iceCoder 调用重叠执行 |
| Eval 模式禁用工具 | 省 ~6K tokens/次，QA 评测不需要文件操作等工具 |
| Eval 模式精简 prompt | 省 ~1K tokens/次，去掉编程助手相关指令 |
| Eval 模式跳过记忆提取 | 省 ~662 次 LLM 调用，QA 回答不应写入新记忆 |

### 关于对话注入方式的说明

本次评测中，对话数据**未通过 WebSocket 逐条发送给 iceCoder**，而是直接以记忆文件的形式写入 `data/memory-files/` 目录。原因如下：

1. **性能瓶颈**：LoCoMo 数据集共 ~5,882 轮对话。通过 WebSocket 发送时，每条消息都会触发完整的 Harness 循环（LLM 调用 + 工具调用 + 记忆提取），实测每条 30-60 秒，10 个样本注入需要 7-15 小时。
2. **成本问题**：逐条注入会产生 ~5,882 次 LLM 调用，仅注入阶段就消耗大量 tokens，而这些调用的目的只是"让系统记住对话"，并非评测目标。
3. **等价性**：iceCoder 的记忆系统最终将对话提取为 Markdown 记忆文件存储在磁盘上。直接写入记忆文件与通过对话注入后由系统自动提取，最终产物格式一致。区别在于跳过了 LLM 提取环节——LLM 提取会做摘要和结构化，而直接写入保留了原始对话全文，信息量更大。
4. **测试聚焦**：本次评测的目标是测试**记忆召回能力**（给定已有记忆，能否正确回答问题），而非记忆写入/提取能力。直接写入记忆文件使测试更纯粹地聚焦于召回环节。

> **注意**：这意味着本次评测结果反映的是 iceCoder 记忆系统在"理想记忆写入"条件下的召回表现。实际使用中，记忆写入依赖 LLM 提取，可能存在信息丢失或摘要偏差，召回表现可能略低于本次评测结果。

### 关于 Token 节省措施的说明

为控制评测成本，启用了以下 Token 节省措施（通过环境变量控制，不影响 iceCoder 正常功能）：

| 措施 | 节省量 | 对测试结果的影响 |
|------|--------|----------------|
| **禁用工具定义** (`ICE_DISABLE_TOOLS=1`) | ~6K tokens/次 × 1,986 次 ≈ 12M tokens | 无影响。QA 评测不需要文件操作、搜索等工具，禁用后模型直接回答问题 |
| **精简 System Prompt** (`ICE_EVAL_MODE=1`) | ~1K tokens/次 × 1,986 次 ≈ 2M tokens | 无影响。去掉了编程助手行为准则、代码规范等与 QA 无关的指令，保留了记忆召回相关指令 |
| **跳过记忆提取** (`ICE_EVAL_MODE=1`) | ~662 次 LLM 调用 ≈ 3.3M tokens | 无影响。QA 的回答不应被写入记忆，跳过提取避免了记忆污染 |
| **每题清空会话** | 每次 input 从 ~150K 降至 ~12K | **正面影响**。消除了对话历史累积导致的"从历史抄答案"问题，使测试更准确地反映记忆召回能力 |
| **Judge 并行评分** | 不省 token，省时间 | 无影响。评分逻辑不变，仅执行方式从串行改为并行 |

**综合效果**：预估总费用从 ~¥110 降至 ~¥14（节省约 87%），同时测试结果更准确。
