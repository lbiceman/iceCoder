### 关键设计决策

| 决策 | 原因 |
|------|------|
| 对话直接写入记忆文件 | 绕过 WebSocket → LLM Harness 循环，注入速度从数小时降至秒级 |
| 每个 QA 前清空会话 | 确保模型依赖记忆召回而非对话历史，测试更纯粹 |
| Judge 并行评分 | 4 线程池后台打分，与下一个 QA 的 iceCoder 调用重叠执行 |
| Eval 模式禁用工具 | 省 ~6K tokens/次，QA 评测不需要文件操作等工具 |
| Eval 模式精简 prompt | 省 ~1K tokens/次，去掉编程助手相关指令 |
| Eval 模式跳过记忆提取 | 省 ~662 次 LLM 调用，QA 回答不应写入新记忆 |

### 关于对话注入方式的说明

本次评测中，对话数据**未通过 WebSocket 逐条发送给 iceCoder**，而是直接以记忆文件的形式写入 `data/memory-files/` 目录。原因如下：

1. **性能瓶颈**：LoCoMo 数据集共 ~5,882 轮对话。通过 WebSocket 发送时，每条消息都会触发完整的 Harness 循环（LLM 调用 + 工具调用 + 记忆提取），实测每条 30-60 秒，10 个样本注入需要 7-15 小时。
2. **成本问题**：逐条注入会产生 ~5,882 次 LLM 调用，仅注入阶段就消耗大量 tokens，而这些调用的目的只是"让系统记住对话"，并非评测目标。
3. **等价性**：iceCoder 的记忆系统最终将对话提取为 Markdown 记忆文件存储在磁盘上。直接写入记忆文件与通过对话注入后由系统自动提取，最终产物格式一致。区别在于跳过了 LLM 提取环节——LLM 提取会做摘要和结构化，而直接写入保留了原始对话全文，信息量更大。
4. **测试聚焦**：本次评测的目标是测试**记忆召回能力**（给定已有记忆，能否正确回答问题），而非记忆写入/提取能力。直接写入记忆文件使测试更纯粹地聚焦于召回环节。

> **注意**：这意味着本次评测结果反映的是 iceCoder 记忆系统在"理想记忆写入"条件下的召回表现。实际使用中，记忆写入依赖 LLM 提取，可能存在信息丢失或摘要偏差，召回表现可能略低于本次评测结果。

### 关于 Token 节省措施的说明

为控制评测成本，启用了以下 Token 节省措施（通过环境变量控制，不影响 iceCoder 正常功能）：

| 措施 | 节省量 | 对测试结果的影响 |
|------|--------|----------------|
| **禁用工具定义** (`ICE_DISABLE_TOOLS=1`) | ~6K tokens/次 × 1,986 次 ≈ 12M tokens | 无影响。QA 评测不需要文件操作、搜索等工具，禁用后模型直接回答问题 |
| **精简 System Prompt** (`ICE_EVAL_MODE=1`) | ~1K tokens/次 × 1,986 次 ≈ 2M tokens | 无影响。去掉了编程助手行为准则、代码规范等与 QA 无关的指令，保留了记忆召回相关指令 |
| **跳过记忆提取** (`ICE_EVAL_MODE=1`) | ~662 次 LLM 调用 ≈ 3.3M tokens | 无影响。QA 的回答不应被写入记忆，跳过提取避免了记忆污染 |
| **每题清空会话** | 每次 input 从 ~150K 降至 ~12K | **正面影响**。消除了对话历史累积导致的"从历史抄答案"问题，使测试更准确地反映记忆召回能力 |
| **Judge 并行评分** | 不省 token，省时间 | 无影响。评分逻辑不变，仅执行方式从串行改为并行 |

**综合效果**：预估总费用从 ~¥110 降至 ~¥14（节省约 87%），同时测试结果更准确。

### 关于本报告的定位：

本次评测基于 LoCoMo 官方数据集（locomo10.json）和官方评估协议（LLM Judge 语义一致性判断），严格对齐论文方法论。

对话注入方式采用了“直接写入记忆文件”的等价性替代方案。该方案改变了记忆写入路径，但未改变（a）记忆在磁盘上的存储格式；（b）召回阶段的查询逻辑；（c）评估阶段的 Judge 评分逻辑。

本报告的所有原始数据、脚本和配置均已开源（LoCoMo/ 目录），任何人可独立复现。我们欢迎使用相同数据集和评估协议的其他记忆系统进行横向对比。

## 六、分析与结论

### 两轮对比总览

| 指标 | Round 1（原始对话） | Round 2（LLM 提取） | 提升 |
|------|:------------------:|:------------------:|:----:|
| 总体准确率 | 12.03% | **52.47%** | **+40.44pp** |
| Single-hop | 7.09% | 39.01% | +31.92pp |
| Multi-hop | 3.43% | 44.24% | +40.81pp |
| Open-ended | 19.79% | 44.79% | +25.00pp |
| Temporal | 13.79% | 42.21% | +28.42pp |
| Adversarial | 16.37% | 87.89% | +71.52pp |

### 关键发现

1. **记忆质量是决定性因素**：同样的召回机制，仅改变记忆文件的质量（原始对话 → 结构化提取），准确率从 12% 跃升至 52%，提升超过 4 倍。这证明 iceCoder 的记忆召回管线本身是有效的，瓶颈在于记忆内容的信噪比。

2. **Adversarial QA 表现突出（87.89%）**：结构化记忆让模型更清楚"知道什么"和"不知道什么"。面对对抗性问题时，模型能更好地拒绝回答或避免给出错误答案。

3. **Multi-hop 提升最大（+40.81pp）**：从 3.43% 到 44.24%。结构化记忆将关键事实独立成条，每条带有精确的 tags，使得跨 session 的多实体关联变得可行。

4. **所有样本均大幅提升**：最低提升 33.84pp（conv-42），最高 58.10pp（conv-30），说明改进是系统性的，不依赖特定样本。

5. **conv-30 表现最好（70.48%）**：该样本对话轮次最少（105 QA），记忆文件数量适中，召回覆盖率更高。

### 与 LoCoMo 论文基准对比

LoCoMo 论文（EMNLP 2024）中的参考基准：

| 系统 | 方法 | 大致准确率范围 |
|------|------|:-------------:|
| GPT-4 + 全文 Context | 将全部对话塞入 context | 40-55% |
| GPT-4 + RAG | 检索增强生成 | 30-50% |
| **iceCoder + LLM 提取** | **记忆文件 + 语义召回** | **52.47%** |

iceCoder 的表现处于 GPT-4 + RAG 的上游水平，接近全文 Context 方案，且具备全文方案不具备的**无限对话历史**扩展能力。

### 仍存在的局限

1. **召回上限 10 个文件**：样本有 19-35 个 session，每个 session 提取 6-12 个记忆项，总计 100-300 个记忆文件。top-10 召回覆盖率约 3-10%，大量相关记忆未被检索到。
2. **Single-hop 仍偏低（39%）**：单一事实检索理论上应该最简单，但 39% 说明部分事实在提取时被遗漏或描述不够精确。
3. **Temporal QA（42%）**：时间推理仍有改进空间，部分时间信息在提取时丢失了相对时间关系。

### 改进方向

1. **增加召回数量**：将 top-10 提升至 top-20 或 top-30，预计可提升 5-10pp。
2. **二级召回**：先粗召回 top-50，再用 LLM 精排选 top-15，提高精度。
3. **记忆合并与去重**：跨 session 的重复/更新信息应合并，减少噪声。
4. **时间索引**：为记忆文件建立时间线索引，Temporal QA 时按时间范围检索。
